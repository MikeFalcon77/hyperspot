## 1. Overview

### 1.1 Purpose

Licensing Service provides a canonical platform API for feature entitlements, effective limits, licensing-aware
decisions, and integration with runtime quota enforcement across ModKit and CyberFabric modules.

The module exists to give platform modules one stable licensing interface regardless of whether the authoritative
licensing source is native ModKit licensing or a legacy VendorA licensing deployment. Licensing Service owns authority
routing, canonical entitlement read models, explanation of entitlement outcomes, and integration with runtime quota
enforcement.

Licensing Service is a control-plane and decision-orchestration module. It is not intended to replace all runtime quota
mechanics in hot paths. Low-latency runtime enforcement remains the responsibility of a quota runtime component that
consumes effective limits from licensing authority.

### 1.2 Background / Problem Statement

The platform needs a licensing capability that supports both new ModKit-native modules and deployments that run inside
legacy VendorA data center environments. Some product scopes are already governed by VendorA licensing systems that own
commercial truth for subscriptions, offering items, usage constraints, and entitlement lifecycle.

Without a canonical licensing facade, every consuming module would need to integrate with legacy licensing directly or
invent local feature-gating and quota interpretation rules. This would create duplicated integration logic, inconsistent
entitlement semantics, and no clean migration path toward a native platform licensing model.

The platform also already needs runtime quota enforcement semantics such as reservation, commit, release, and
reconciliation. Those mechanics are valuable, but they must not become a second business source of truth for commercial
licensing. Licensing Service therefore acts as the canonical authority layer above runtime quota enforcement.

### 1.3 Goals (Business Outcomes)

- Provide one canonical licensing API and SDK for all ModKit and CyberFabric modules.
- Support federated authority so legacy VendorA licensing can remain authoritative for selected scopes during migration.
- Normalize legacy and native licensing concepts into a canonical platform model.
- Expose feature entitlement, effective limit, and explanation data without forcing consumers to understand legacy
  domain quirks.
- Integrate with runtime quota enforcement without making quota runtime the owner of commercial licensing truth.
- Enable gradual migration from legacy-authoritative to native-authoritative licensing.

| Term | Definition |

| Offering Item | The commercial or operational unit that bundles one or more enabled features and limit semantics. |
| Entitlement Grant | A canonical record that a scope is allowed to use an offering item or feature subject to policy and limit state. |

### 3.1 Module-Specific Environment Constraints

- The module must support deployments both inside legacy VendorA data center environments and in platform-native environments.
- Hot-path licensing decisions must rely on local projections, caches, or native stores rather than deep synchronous traversal of legacy authority graphs.
- Degraded behavior must preserve explicit authority identity, freshness state, and limiting-factor information when upstream authority data is stale or unavailable.

## 4. Scope

### 4.1 In Scope

- Canonical licensing API and SDK for platform consumers.
- Authority routing between legacy-authoritative and native-authoritative scopes.
- Canonical read model for feature entitlements, offering items, and effective limits.
- Legacy VendorA licensing adapter as a ModKit plugin boundary behind the Licensing Service public API and normalization boundary.
- Native licensing authority mode for platform-native scopes.
- Decision explanation that identifies authority, limiting factor, and policy basis.
- Integration with runtime quota enforcement for effective-limit-aware decisions.
- Shadow-mode comparison support during migration.
- Licensing-health observability, reconciliation hooks, and sync status visibility.

### 4.2 Out of Scope

- License key generation and serial-number pool operations; owned by legacy VendorA licensing or a future dedicated licensing backoffice.
- Billing, invoicing, taxation, and payment processing; owned by billing and commercial systems.
- Full runtime reservation lifecycle semantics; owned by quota runtime as specified in `modules/quota-service/docs/PRD.md`.
- Direct per-module legacy API integrations; forbidden by architecture because modules must use the licensing-service SDK.
- Detailed reseller, distributor, OEM, and perpetual-license backoffice workflows unless explicitly adopted later.
- Generic feature flag management unrelated to licensing or entitlements.

### 4.3 Relationship to Quota Service

The current `modules/quota-service/docs/PRD.md` does **not** move wholesale into this PRD.

Instead, the relationship is:
- Licensing Service owns business entitlement truth, authority routing, and effective-limit resolution.
- Quota runtime owns low-latency runtime enforcement semantics such as `Check`, `CheckAndReserve`, `Commit`, `Release`, reservation state, and reconciliation of runtime enforcement state.
- Licensing Service provides effective limits and entitlement context to quota runtime.
- Quota runtime must not independently redefine commercial licensing truth for a scope that is already authoritative in licensing.

This means the quota PRD becomes a **related runtime specification**, not a replacement for the new licensing PRD.

### 5.1 Canonical API and Domain Model

#### Canonical Licensing API

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-canonical-api`

The system **MUST** provide a canonical API and SDK for feature entitlement checks, effective limit lookup, effective
entitlement inspection, and licensing-aware quota decision orchestration.

- **Rationale**: Modules need one stable integration surface that remains valid across legacy-authoritative and native-authoritative deployments.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-quota-runtime`

#### Canonical Domain Model

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-canonical-model`

The system **MUST** expose a canonical domain model that includes feature, offering item, entitlement grant, effective
limit, authority, and decision explanation concepts without leaking raw legacy VendorA payloads into consuming modules.

- **Rationale**: Consumers must not couple to authority-specific schemas or legacy business quirks.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-tenant-administrator`

#### Decision Contract

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-decision-contract`

The system **MUST** define a canonical decision contract for all decision-producing licensing operations. Each produced
decision **MUST** include at minimum `decision_outcome`, `decision_state`, `resolved_authority_mode`, normalized licensing scope identifier, `policy_effect`, primary `reason_code`, structured explanation, `correlation_id`, `freshness_state`, and decision timestamp.

- **Rationale**: A stable decision contract is required so multiple modules and reviewers do not infer different semantics from the same response.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-quota-runtime`, `cpt-cf-licensing-service-actor-platform-operator`

#### Decision Outcome Semantics

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-decision-outcome-semantics`

The system **MUST** use `decision_outcome = allow` only when the licensing service determines that the request is
permitted by entitlement and applicable limit policy, and **MUST** use `decision_outcome = deny` only when the
licensing service determines that the request is not permitted. Decision outcomes **MUST NOT** encode invalid request,
routing failure, or internal processing failure semantics. `capped`, `downgraded`, and `deferred` **MUST NOT** be
represented as decision outcomes; they **MUST** be represented only through `policy_effect` while preserving
`decision_outcome = allow` or `decision_outcome = deny`. `policy_effect = capped`, `downgraded`, or `deferred` **MUST** be returned only with `decision_outcome = allow`. `decision_outcome = deny` **MUST** use `policy_effect = none`.

- **Rationale**: Consumers need stable business meaning for allow and deny without overloading outcomes to represent transport or processing failures.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-quota-runtime`

#### Lookup Response Contract Semantics

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-lookup-contract`

The system **MUST** define canonical response contracts for `GetEffectiveEntitlements` and `GetEffectiveLimits`. Each
successful lookup response **MUST** either embed the canonical decision contract or expose an equivalent response
contract that includes at minimum normalized licensing scope identifier, `resolved_authority_mode`, `freshness_state`,
`decision_state` when degraded semantics apply, and `correlation_id`. Lookup responses **MUST** preserve explicit
separation between successful responses and service errors. A lookup response served under degraded policy **MUST NOT**
imply fresh authoritative data.

- **Rationale**: Read-only lookup operations must not force consumers to guess whether freshness, degraded handling, and error semantics differ from decision-producing operations.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-quota-runtime`, `cpt-cf-licensing-service-actor-platform-operator`

#### Reason and Error Semantics

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-reason-error-semantics`

The system **MUST** use documented closed-set machine-readable codes for both decision reasons and service errors.
Decision `reason_code` values **MUST** include at least `feature_not_entitled`, `offering_item_missing`,
`limit_exceeded`, `scope_disabled`, `policy_denied`, `authority_data_stale`, and `quota_rejected`. Service
`error_code` values **MUST** include at least `invalid_request`, `scope_unresolved`,
`conflicting_authority_configuration`, `authority_unavailable`, and `internal_failure`.

- **Rationale**: Fixed codes are required for reliable downstream handling, audit analysis, and cross-team interoperability.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-migration-engineer`

### 5.2 Federated Authority Routing

#### Normalized Licensing Scope

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-normalized-scope`

The system **MUST** derive a `Normalized Licensing Scope` for every decision-producing operation and every effective
entitlement or effective-limit lookup. The normalized scope **MUST** use the canonical tuple
`(tenant_scope, subject_scope, product_family, offering_item_or_feature, deployment_environment, usage_dimension)`.
The same normalized scope tuple **MUST** be used for authority routing, cache identity, audit records, and shadow
comparison for the same logical request.

- **Rationale**: Routing, caching, auditing, and migration comparison are unsafe if different teams normalize the same request differently.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-migration-engineer`

#### Explicit Authority Mode per Licensing Scope

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-authority-routing`

The system **MUST** resolve an explicit authority mode for each licensing scope and **MUST** support at least
`legacy_authoritative`, `native_authoritative`, `shadow`, and `disabled` modes. Authority routing **MUST** be
deterministic for the same normalized licensing scope input. If no authority mode can be resolved, the system **MUST**
return a service error rather than implicitly selecting an authority. For the same normalized licensing scope, routing
**MUST NOT** depend on caller-local normalization or undocumented defaults.

- **Rationale**: Hybrid deployments require deterministic routing and migration behavior rather than implicit fallback.
- **Actors**: `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-migration-engineer`

#### Single Authority per Scope

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-single-authority`

The system **MUST NOT** allow the same licensing scope to be simultaneously authoritative in both legacy and native
modes.

- **Rationale**: A scope must have exactly one business source of truth to avoid conflicting entitlement decisions.
- **Actors**: `cpt-cf-licensing-service-actor-platform-operator`

#### Disabled Mode Semantics

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-disabled-mode`

For any decision-producing operation whose resolved authority mode is `disabled`, the system **MUST** return a
canonical decision with `decision_outcome = deny`, `decision_state = authoritative`, `resolved_authority_mode =
disabled`, `policy_effect = none`, and primary `reason_code = scope_disabled`. The system **MUST NOT** convert a
resolved `disabled` mode into a service error unless the request itself is invalid.

- **Rationale**: `disabled` must have one stable product meaning so callers do not interpret it as either business denial or infrastructure failure depending on team implementation.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-platform-operator`

#### No Cross-Authority Failover

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-no-cross-authority-failover`

After resolving the authority mode for a licensing scope, the system **MUST NOT** switch the request to a different
authority because the resolved authority is stale, slow, or unavailable. Cross-authority substitution is forbidden;
the system may return only a service error or a policy-permitted degraded decision derived from the resolved authority
context.

- **Rationale**: Silent cross-authority fallback would create hidden dual-truth behavior and unsafe migration outcomes.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-platform-operator`

#### Freshness and Degraded Decision Semantics

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-freshness-semantics`

The system **MUST** classify authority data freshness using the closed set `fresh`, `stale_within_policy`, and
`stale_beyond_policy`. A decision with `decision_state = authoritative` **MUST** be based only on `fresh` authority
data. A decision with `decision_state = degraded` **MUST** be returned only when product policy explicitly permits use
of `stale_within_policy` data or another non-authoritative evaluation path. `stale_beyond_policy` data **MUST NOT** be
represented as authoritative.

- **Rationale**: Freshness semantics must be explicit so consumers can distinguish valid authoritative outcomes from tolerated degraded outcomes.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-migration-engineer`

#### Degraded Eligibility by Operation Class

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-degraded-policy`

The system **MUST** apply degraded-decision policy by operation class. `CheckFeatureEntitlement`,
`GetEffectiveEntitlements`, `GetEffectiveLimits`, and `GetLicensingDecision` are the only operation classes that may
return `decision_state = degraded`, and only when product policy explicitly marks the normalized licensing scope as
degraded-eligible for that operation class. Degraded eligibility and freshness thresholds **MUST** be defined for each
product family before implementation of that product family and **MUST** be evaluated using the same normalized
licensing scope and operation class as the live request. State-changing entitlement-management operations and
authority-configuration operations **MUST NOT** return degraded success results; when fresh authoritative evaluation is
unavailable they **MUST** return service errors.

- **Rationale**: Read-only decision paths and state-changing operations need different safety rules under stale or unavailable authority data.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-native-licensing-store`

#### Authority-Aware Explanation

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-explanation`

The system **MUST** return explanation metadata that identifies the resolved authority, the limiting authority
component, the policy or entitlement basis for each decision outcome, the primary `reason_code`, zero or more
contributing reason codes, the `decision_state`, the `policy_effect`, the `freshness_state`, and whether the limiting
factor originated in licensing entitlement evaluation or runtime quota evaluation.

- **Rationale**: Operators and integrators need to understand whether a denial or restriction came from legacy entitlement, native entitlement, or runtime quota enforcement.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-tenant-administrator`

#### Error and Degraded Decision Distinction

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-error-distinction`

The system **MUST** distinguish service errors from decision objects. A service error **MUST** be returned when no
semantically valid licensing decision can be produced. A degraded decision **MUST** still conform to the canonical
decision contract, **MUST** carry `decision_state = degraded`, and **MUST** include the reason that non-authoritative
evaluation was used.

- **Rationale**: Consumers and reviewers must not confuse degraded business decisions with failures to produce any decision at all.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-quota-runtime`, `cpt-cf-licensing-service-actor-platform-operator`

### 5.3 Legacy VendorA Integration

#### Anti-Corruption Translation

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-legacy-adapter`

The system **MUST** implement the legacy VendorA adapter as a ModKit plugin behind the Licensing Service main module.
The adapter plugin **MUST** translate authority-specific concepts into the canonical licensing model before responses

- **Rationale**: Direct exposure of legacy VendorA structures would spread coupling and hinder future migration.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-legacy-vendora-licensing`

#### Plugin Isolation for Authority Adapters

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-adapter-plugin-isolation`

The system **MUST** keep authority-adapter implementation selection internal to Licensing Service. Any legacy or
future authority adapter plugin **MUST** be discoverable and selectable only by Licensing Service using ModKit plugin
mechanisms, and **MUST NOT** create a separate consumer-facing contract that bypasses the Licensing Service public API,
canonical routing rules, decision contract, or audit semantics.

- **Rationale**: Plugin isolation preserves swappable authority implementations without exposing vendor-specific behavior or bypassing canonical product semantics.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-legacy-vendora-licensing`

#### Legacy Projection Read Model

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-legacy-projection`

The system **MUST** maintain a normalized read model or equivalent projection for legacy-authoritative entitlement and
effective-limit data suitable for low-latency read access by the licensing service.

- **Rationale**: Hot-path licensing decisions should not require synchronous deep legacy graph traversal on every request.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-quota-runtime`, `cpt-cf-licensing-service-actor-legacy-vendora-licensing`

#### Sync Status Visibility

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-sync-status`

The system **MUST** expose synchronization status, source version or watermark metadata, and staleness indicators for
legacy-authoritative projections.

- **Rationale**: Operators need visibility into projection freshness and migration health.
- **Actors**: `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-migration-engineer`

- **Rationale**: Native-authoritative scopes require first-class management rather than adapter-only behavior.

### 5.5 Runtime Quota Integration

#### Effective Limit Delivery to Runtime Quota

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-quota-runtime-integration`

The system **MUST** provide effective limit and entitlement context to the runtime quota component so hot-path quota
decisions can be made without re-implementing licensing authority logic inside quota runtime.

- **Rationale**: Runtime quota enforcement must consume licensing truth, not duplicate it.
- **Actors**: `cpt-cf-licensing-service-actor-quota-runtime`

#### Distinct Ownership Between Licensing and Runtime Usage

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-usage-ownership`

The system **MUST** distinguish between business-authoritative licensing usage state and runtime operational reservation
or counter state and **MUST** document which system is authoritative for each supported usage class. At minimum,
the following ownership matrix **MUST** apply:

- `entitlement_existence`: Licensing Service
- `feature_enablement`: Licensing Service
- `effective_licensed_limit`: Licensing Service
- `commercially_authoritative_consumed_quantity`: Licensing Service
- `runtime_reserved_quantity`: Quota Runtime
- `runtime_inflight_or_window_counter`: Quota Runtime

Runtime-owned quantities **MUST NOT** mutate licensing truth unless a product-specific extension to this PRD explicitly
reclassifies that usage class before implementation for that product family.

- **Rationale**: Hybrid deployments fail when runtime counters and commercial usage both claim authority over the same quantity.
- **Actors**: `cpt-cf-licensing-service-actor-platform-operator`, `cpt-cf-licensing-service-actor-migration-engineer`, `cpt-cf-licensing-service-actor-quota-runtime`

#### Licensing-Aware Decision Orchestration

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-decision-orchestration`

The system **MUST** support orchestration flows where licensing entitlement evaluation and runtime quota evaluation
participate in one explainable decision outcome. When both participate, the returned decision **MUST** identify which
component produced the final limiting condition and **MUST NOT** obscure whether the final deny or restriction was caused
by entitlement state or by quota runtime state.

- **Rationale**: Consumers need one coherent decision even when commercial entitlement and runtime usage are governed by separate subsystems.
- **Actors**: `cpt-cf-licensing-service-actor-platform-module`, `cpt-cf-licensing-service-actor-quota-runtime`

### 5.6 Migration and Shadowing

#### Shadow Comparison Mode

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-shadow-mode`

The system **MUST** support shadow mode in which non-authoritative native decisions are computed and compared against
authoritative legacy outcomes without affecting live decisions. In `shadow` mode, the authoritative decision **MUST** be
the only user-visible decision, and the shadow decision **MUST NOT** alter the returned allow or deny outcome,
effective-limit context, or service-error semantics for the live request.

- **Rationale**: Migration requires safe comparison before cutover.
- **Actors**: `cpt-cf-licensing-service-actor-migration-engineer`, `cpt-cf-licensing-service-actor-platform-operator`

#### Mismatch Visibility

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-fr-shadow-observability`

The system **MUST** record and expose mismatches between authoritative and shadow decisions with enough context to debug
model or mapping differences. Each mismatch record **MUST** include a `correlation_id`, authoritative and shadow
reason codes, authoritative and shadow decision states, and a `shadow_mismatch_class` from the closed set
`outcome_mismatch`, `reason_mismatch`, `limit_mismatch`, `missing_shadow_decision`, and `missing_authoritative_decision`.
Shadow comparison **MUST** be performed only across the same normalized licensing scope, the same usage dimension, and
canonicalized effective-limit units. `limit_mismatch` **MUST** be recorded only when both authoritative and shadow
decisions produce effective limits for the same canonical usage dimension but disagree on the canonical value or
`policy_effect`.

- **Rationale**: Shadowing is only useful if divergences are measurable and actionable.
- **Actors**: `cpt-cf-licensing-service-actor-migration-engineer`, `cpt-cf-licensing-service-actor-platform-operator`

## 6. Non-Functional Requirements

> **Global baselines**: Project-wide NFRs (performance, security, reliability, scalability) defined in root PRD and [guidelines/](../../../guidelines/). Document only module-specific NFRs here: **exclusions** from defaults or **standalone** requirements.
>
> **Testing strategy**: NFRs verified via automated benchmarks, security scans, and monitoring unless otherwise specified.

### 6.1 Module-Specific NFRs

#### Decision Latency

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-decision-latency`

The system **MUST** complete canonical entitlement lookup and effective-limit lookup operations within 50 ms at p95 for
warm-path requests backed by a fresh local projection or native store.

- **Threshold**: `<= 50 ms p95` for warm-path read operations using fresh local authority data.
- **Rationale**: Hot-path gating and quota orchestration require low-latency entitlement resolution.
- **Architecture Allocation**: See DESIGN.md § NFR Allocation.

#### Decision Availability

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-availability`

The system **MUST** provide deterministic outcomes for 99.9% of monthly licensing decision requests, including explicit
degraded decisions when policy permits non-authoritative evaluation and documented service errors when a valid decision
cannot be produced.

- **Threshold**: `99.9%` of monthly externally visible decision-producing requests at the licensing-service boundary return either authoritative decisions, policy-defined degraded decisions, or documented service errors with fixed `error_code` semantics. Internal retries, projection refresh operations, and shadow sub-requests are excluded from this denominator.
- **Rationale**: Platform modules require predictable behavior even during authority outages or projection lag.
- **Architecture Allocation**: See DESIGN.md § NFR Allocation.

#### Observability

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-observability`

The system **MUST** emit structured telemetry for authority routing, legacy projection freshness, native decisioning,
shadow mismatches, and quota-runtime orchestration outcomes.

- **Threshold**: All decision-producing operations emit `correlation_id`, authority mode, decision outcome, decision state, primary reason code, freshness metadata, and limiting-factor metadata; all shadow comparisons emit `shadow_mismatch_class` when a mismatch exists.
- **Rationale**: Operators and migration engineers need consistent visibility into routing behavior and mismatch causes.
- **Architecture Allocation**: See DESIGN.md § NFR Allocation.

#### Auditability

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-auditability`

The system **MUST** retain enough decision and routing metadata to reconstruct why an entitlement or limit decision was
produced for at least 30 days in production environments.

- **Threshold**: Decision records retained for `>= 30 days` include at minimum `correlation_id`, licensing scope, resolved authority mode, decision outcome, decision state, primary reason code, freshness state, and source version or watermark when available.
- **Rationale**: Tenant support, compliance investigations, and migration validation need post-factum decision reconstruction.
- **Architecture Allocation**: See DESIGN.md § NFR Allocation.

#### Correlation and Traceability

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-correlation-traceability`

The system **MUST** propagate the same `correlation_id` across decision responses, service-error responses, audit
records, observability events, and shadow comparison records generated by the same logical request.

- **Threshold**: `100%` of decision-producing requests emit one stable `correlation_id` across response and emitted records.
- **Rationale**: Multi-system review and incident analysis require end-to-end traceability for a single licensing decision.
- **Architecture Allocation**: See DESIGN.md § NFR Allocation.

#### Single-Authority Safety

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-no-double-truth`

The system **MUST** enforce configuration and validation rules that prevent one licensing scope from having more than
one authoritative source at the same time.

- **Threshold**: All configuration changes that would create dual-authoritative scope ownership are rejected.
- **Rationale**: Conflicting business truth creates inconsistent entitlements and unsafe migration behavior.
- **Architecture Allocation**: See DESIGN.md § NFR Allocation.

#### Staleness Visibility

- [ ] `p1` - **ID**: `cpt-cf-licensing-service-nfr-staleness-visibility`

The system **MUST** expose explicit stale-state signals and apply the configured degraded-mode policy rather than
silently serving stale data as fresh when projection or cache staleness exceeds configured policy thresholds.

- **Threshold**: Any response served after freshness threshold breach includes explicit stale-state metadata and degraded-mode semantics.
- **Rationale**: Consumers and operators must be able to distinguish fresh authoritative data from tolerated stale reads.
- **Architecture Allocation**: See DESIGN.md § NFR Allocation.

- No module-specific exclusions identified; project-default NFR baselines apply unless explicitly extended above.

## 8. Use Cases

Optional: Include when interaction flows add clarity beyond requirement statements.

#### Check Feature Entitlement for a Module Request

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-usecase-check-feature-entitlement`

**Actor**: `cpt-cf-licensing-service-actor-platform-module`

**Preconditions**:
- Licensing scope is derivable from request context.
- Licensing authority data is available from either a fresh projection or native store.

**Main Flow**:
1. The platform module calls the canonical licensing API with subject, feature, and scope context.
2. The licensing service resolves authority mode for the scope.
3. The licensing service loads canonical entitlement and effective-limit context.
4. The licensing service returns a decision with `decision_outcome`, `decision_state`, `policy_effect`, explanation,
   freshness metadata, and `correlation_id`.

**Postconditions**:
- The platform module receives an explainable licensing decision.

**Alternative Flows**:
- **Authority data stale or unavailable**: The licensing service returns a policy-defined degraded or denied outcome with stale-state metadata.
- **Invalid request or unresolved scope**: The licensing service returns a service error with fixed `error_code` semantics instead of a business decision.

#### Provide Effective Limits to Runtime Quota

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-usecase-provide-effective-limits`

**Actor**: `cpt-cf-licensing-service-actor-quota-runtime`

**Preconditions**:
- The runtime quota component has a resolved licensing scope.
- Effective-limit data is available from authoritative or policy-allowed degraded sources.

**Main Flow**:
1. Quota runtime requests effective-limit and entitlement context from the licensing service.
2. The licensing service resolves authority mode and returns canonical effective-limit data.
3. Quota runtime applies `Check`, `CheckAndReserve`, `Commit`, or `Release` semantics using licensing-derived limits while preserving limiting-component attribution from the licensing response.

**Postconditions**:
- Quota runtime has authoritative limit context for runtime enforcement.

**Alternative Flows**:
- **No entitlement grant exists**: The licensing service returns explicit no-entitlement context and runtime quota denies or caps the operation accordingly.
- **Licensing decision unavailable without degraded policy**: The licensing service returns a service error rather than silently substituting another authority.

#### Compare Shadow Decisions During Migration

- [ ] `p2` - **ID**: `cpt-cf-licensing-service-usecase-compare-shadow-decisions`

**Actor**: `cpt-cf-licensing-service-actor-migration-engineer`

**Preconditions**:
- The licensing scope is configured in `shadow` mode.
- Both authoritative legacy and shadow native decision paths are available.

**Main Flow**:
1. A licensing decision request is processed using the authoritative legacy path.
2. The licensing service computes the shadow native decision for the same input.
3. The licensing service stores comparison output, mismatch classification, and `correlation_id` metadata.
4. The migration engineer reviews mismatches and determines readiness for cutover.

**Postconditions**:
- Migration analysis data is available without changing live decision outcomes.

**Alternative Flows**:
- **Shadow path unavailable**: The licensing service records incomplete comparison metadata without affecting the authoritative live decision.

## 9. Acceptance Criteria

Business-level acceptance criteria for the PRD as a whole.

- [ ] A decision-producing licensing operation returns a canonical decision object containing `decision_outcome`, `decision_state`, resolved authority mode, normalized licensing scope, `policy_effect`, primary `reason_code`, explanation, `freshness_state`, and `correlation_id`.
- [ ] `policy_effect = capped`, `downgraded`, and `deferred` are returned only with `decision_outcome = allow`, and `decision_outcome = deny` always uses `policy_effect = none`.
- [ ] Invalid request, unresolved scope, conflicting authority configuration, and unavailable authority without permitted degraded handling return service errors with fixed `error_code` semantics instead of business decisions.
- [ ] A licensing scope cannot be configured as authoritative in both legacy and native modes simultaneously, and no request silently falls back from one authority to another after routing has been resolved.
- [ ] Legacy-authoritative and native-authoritative scopes can coexist in the same deployment through explicit routing, and the same normalized licensing scope resolves to the same authority mode deterministically across routing, cache identity, audit records, and shadow comparison.
- [ ] When `resolved_authority_mode = disabled`, decision-producing operations return `decision_outcome = deny` with primary `reason_code = scope_disabled` rather than a service error.
- [ ] Degraded decisions are explicitly marked `decision_state = degraded`, include stale or degraded reason semantics, and are never reported as authoritative.
- [ ] Only `CheckFeatureEntitlement`, `GetEffectiveEntitlements`, `GetEffectiveLimits`, and `GetLicensingDecision` may return degraded decisions, and state-changing entitlement-management or authority-configuration operations return service errors instead of degraded success results when fresh authoritative evaluation is unavailable.
- [ ] `GetEffectiveEntitlements` and `GetEffectiveLimits` return either the canonical decision contract or an equivalent response contract containing normalized licensing scope, resolved authority mode, freshness metadata, degraded-state metadata when applicable, `correlation_id`, and separate service-error semantics.
- [ ] Decision responses identify the resolved authority and limiting component clearly enough to distinguish entitlement-driven denial from quota-driven denial or restriction.
- [ ] The legacy VendorA adapter is implemented as a ModKit plugin behind the Licensing Service main module, and platform consumers cannot consume the adapter plugin directly or bypass Licensing Service public contracts.
- [ ] Runtime quota enforcement consumes licensing-derived entitlement and effective-limit context without taking ownership of `entitlement_existence`, `feature_enablement`, `effective_licensed_limit`, or `commercially_authoritative_consumed_quantity`.
- [ ] Shadow mode compares authoritative and shadow decisions for the same normalized scope, canonical usage dimension, and canonical units, preserves the live authoritative response, and records mismatch classifications with `correlation_id` metadata.
- [ ] Audit and telemetry records for a decision can be joined by `correlation_id` and show authority mode, decision outcome, decision state, primary reason code, and freshness state.

## 11. Assumptions

- Some production environments will continue to depend on legacy VendorA licensing for selected scopes during transition.
- Not all modules need native-authoritative licensing on day one.
- Runtime quota semantics remain useful and should be preserved rather than discarded.
- The platform can maintain canonical projections or cached views of authority data.
- Operators can manage authority-routing configuration per licensing scope.

| Risk | Impact | Mitigation |
